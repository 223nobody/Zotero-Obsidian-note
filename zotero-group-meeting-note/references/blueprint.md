# Blueprint

用这份蓝图来起草中文计算机论文组会分享笔记。蓝图的目标不是把论文机械拆成固定小节，而是确保读者能讲清楚论文的核心问题、创新点、证据链，以及每一张图、每一个表、每一个公式到底说明了什么。

## 核心原则

组会笔记必须服务于“讲清楚”，而不只是“总结过”。最终笔记至少要回答下面几个问题：

- 这篇论文到底解决了什么问题，为什么值得做？
- 它相对已有工作的真正增量在哪里？
- 图、表、公式、实验数据分别在支撑什么结论？
- 结论成立的前提、边界和证据缺口是什么？
- 它在相关工作脉络中处于什么位置？
- 组会上最值得追问的讨论点是什么？

## 术语与语言润色规则

- 默认优先用自然中文讲清楚机制和结论，不要让整段文字被英文术语堆满。
- 术语润色时参考 `references/terminology.md`；如果有论文全文或 Markdown，优先先用 `scripts/extract_terms.py <source-file>` 扫一遍高频英文候选词。
- 对高频、关键、专业性的英文短语，在首次关键出现处写成 `English phrase（中文翻译）`。
- 同一个术语全篇保持一个主译法；后文可以用稳定简称，但不要来回切换不同中文译名。
- 数据集、模型名、Benchmark、指标缩写、API 名、作者自定义模块名如果翻译后会降低可检索性，则保留英文原名，并补一句简短中文解释。
- 如果缩写是论文中的主称呼，可以直接写成 `Acronym（中文翻译）`，例如 `MORL（多目标强化学习）`。
- 术语翻译要服务于组会可讲性，不要把所有普通英文都机械翻译，只处理真正影响理解的专业短语。
- 最后做一次口语化润色：把过长句子拆开，把抽象判断落到证据或例子上，保留技术准确性。

## 写作前置规则

在开始写组会笔记正文之前，首先保证执行以下步骤，不能跳过：

- 先阅读当前 md 格式论文或论文抽取后的 Markdown 全文。
- 首先根据 MinerU 的论文数据源,将论文中的所有图片保存到对应 Obsidian 笔记同级目录下的 `assets/` 文件夹。
- 图片命名尽量可读且可定位，例如 `figure-1.png`、`figure-2-method-overview.png`、`equation-3.png`。
- 后续写图片解释、公式分析和证据链时，优先引用已经保存到 `assets/` 的本地图片。

## 强制覆盖规则

默认要求对论文中出现的所有图片、表格、公式给出对应解释，不能只挑关键的几项来写。

### 0. 原文顺序硬规则

- `## 五、图片解释` 内的所有 Figure 条目必须按原论文中首次出现的顺序排列。
- `## 六、表格与结论数据解释` 内的所有 Table 条目必须按原论文中首次出现的顺序排列。
- `## 七、公式解释` 内的所有 Equation / Loss / Objective / Score / Constraint 条目必须按原论文中首次出现的顺序排列。
- 这条规则只约束排列顺序；除非内容有事实错误、证据缺口或格式损坏，不要因为排序而改写条目内容和模板。
- 如果抽取文本无法确认首次出现位置，按论文编号顺序作为 fallback，并在自检时标记 `顺序依据：编号推断`。

### 1. 覆盖粒度

- 所有 `Figure` 都要解释；如果同一大图包含多个子图，至少要交代每个子图分别在看什么。
- 所有 `Table` 都要解释；至少说明比较对象、指标和主要结论。
- 所有关键 `Equation` / `Loss` / `Objective` / `Score` / `Constraint` 都要解释；纯符号定义式可以合并解释，但不能完全跳过。

### 2. 压缩方式

如果论文里的图、表、公式很多，可以压缩写法，但不能省掉编号：

- 关键项：使用“完整解释模板”。
- 次要项：使用“压缩解释模板”。
- 同类连续项：可以合并成一个小节，例如“Table 4-6 共同说明泛化与效率”，但要逐一提到编号和差异。

### 3. 缺失处理

- 如果当前材料里看不到对应内容，明确写 `待核对原文：Figure X / Table Y / Equation Z`。
- 如果数值只能从图中目测，不要写成精确事实，要标注为“目测推断”。
- 如果某张图、某个表、某个公式并没有真正支撑作者的主张，要直接指出“证据不足”或“更多是辅助说明”。

## 先判断论文类型

先判断论文主类型，因为这会决定你优先解释什么证据。

| 论文类型              | 最该讲清楚的东西       | 优先证据                                  |
| --------------------- | ---------------------- | ----------------------------------------- |
| 方法类论文            | 新方法为什么有效       | 方法图、关键公式、主实验、消融            |
| 系统类论文            | 系统设计如何解决约束   | 架构图、流程图、吞吐/延迟/成本/可靠性结果 |
| 数据集/Benchmark 论文 | 数据或评测为什么有价值 | 数据组成、标注协议、任务定义、基线结果    |
| 分析类论文            | 发现了什么规律或缺陷   | 统计图、分组实验、案例分析、反例          |
| 理论类论文            | 命题、假设和证明直觉   | 核心公式、定理条件、证明结构、适用范围    |
| 应用类论文            | 技术如何落地到场景     | 场景约束、实际效果、失败案例、部署成本    |

如果一篇论文同时有多种属性，先确定一个主类型，再用 1-2 句话交代副类型。

## 推荐输出结构

可以按下面的结构组织，也可以按论文复杂度合并小节，但不要破坏“问题 - 创新 - 机制 - 证据 - 边界”的主线。

```markdown
# 组会分享笔记：<论文标题>

## 论文基本信息

## 一、核心结论

## 二、问题背景与研究定位

## 三、创新点分析

## 四、方法或系统机制

## 五、图片解释

## 六、表格与结论数据解释

## 七、公式解释

## 八、证据链与可信度

## 九、与相关工作的对比与联系

## 十、优点、局限与讨论问题

## 十一、5-8 分钟汇报提纲
```

## 各部分写法要求

### 论文基本信息

建议保留：

| 项目        | 内容                                                                      |
| ----------- | ------------------------------------------------------------------------- |
| 论文标题    | <title>                                                                   |
| 作者 / 机构 | <authors / affiliations>                                                  |
| 发表信息    | <venue / year / DOI / arXiv>                                              |
| 论文类型    | <method / system / benchmark / dataset / analysis / theory / application> |
| 研究方向    | <LLM / NLP / CV / Systems / Security / DB / HCI / ML / Other>             |

### 一、核心结论

用 3-6 句话先给出“结论版摘要”，至少覆盖：

- 论文在解决什么问题。
- 核心方法或系统想法是什么。
- 最关键的证据来自哪几张图、哪几个表、哪几个公式。
- 结论成立的前提和边界是什么。
- 组会上最值得讨论的一点是什么。

### 二、问题背景与研究定位

只写对理解创新点有帮助的背景，不要铺得太长。至少回答：

- 旧路线一般怎么做。
- 旧路线卡在哪里。
- 本文到底打算改哪一层：数据、模型、训练目标、推理流程、系统架构、评测协议，还是理论假设。

### 三、创新点分析

不要照搬论文里的 contribution list。每个创新点都要和“旧问题 - 新做法 - 证据 - 边界”连起来写。

每个创新点至少回答：

- 新在什么地方。
- 相对哪条已有路线是新的。
- 为什么这点重要。
- 被什么证据支撑。
- 在什么条件下可能不成立。

可选表格：

| 创新点       | 对应旧问题         | 支撑证据                          | 可信边界   |
| ------------ | ------------------ | --------------------------------- | ---------- |
| <innovation> | <prior limitation> | <figure/table/formula/experiment> | <boundary> |

### 四、方法或系统机制

重点讲清机制，不要求把论文所有细节重写一遍。优先说明：

- 整体流程长什么样。
- 真正承载创新的模块在哪里。
- 输入、输出和中间状态如何变化。
- 为什么这种设计理论上或经验上合理。
- 哪些部分是核心贡献，哪些只是实现细节。

如果论文有总览图、系统图或流程图，优先用它建立全局视角。

## 图片、表格、公式的解释规则与模板

下面三类内容是这份蓝图的重点。默认逐项覆盖，不允许只贴标题或只抄 caption。
写作时先根据 `references/source-order.md` 建立 Figure / Table / Equation 顺序清单，再分别填入下面三个章节；不要按重要性或叙事便利重新排序。

### 五、图片解释

#### 图片解释总要求

请你按照论文顺序解释所有图片，每张图至少要交代：

- 图里在展示什么对象、过程或现象。
- 图中关键元素分别代表什么。
- 这张图和哪一个创新点或结论有关。
- 这张图能证明到什么程度，不能证明什么。
- 适合组会口头怎么讲。

#### 图片完整解释模板

```markdown
#### Figure <编号>：<图题或自拟短标题>

**图意概述**  
这张图展示了 <对象 / 流程 / 现象>，核心是在说明 <一句话结论>。

**图中元素说明**

- <元素 1> 表示 <含义>
- <元素 2> 表示 <含义>
- <颜色 / 箭头 / 区域 / 坐标轴> 表示 <含义>

**作者想证明什么**  
作者希望借这张图支撑 <创新点 / 机制解释 / 实验结论>。

**我对这张图的解读**  
从图中可以读出 <趋势 / 对比 / 模块关系 / 关键现象>。真正重要的不是图长什么样，而是它说明了 <为什么方法有效 / 为什么系统可行 / 为什么结论成立>。

**证据边界或风险**  
这张图的解释边界在于 <样本选择 / 坐标缩放 / 可视化方式 / 是否只展示成功案例 / 是否缺少误差条等>，因此它更像 <强证据 / 中等证据 / 辅助证据>。

**组会讲法**  
“Figure <编号> 可以先看成在回答 <问题>。图里的 <关键元素> 对应 <含义>，最重要的现象是 <现象>。所以它支持的其实是 <结论>，但这个结论还受到 <边界> 的限制。”
```

#### 图片压缩解释模板

```markdown
- **Figure <编号>**：展示 <对象>。重点现象是 <现象>，对应支撑 <创新点/结论>。限制在于 <边界>。
```

#### 按图片类型补充关注点

| 图片类型        | 额外要点                                  |
| --------------- | ----------------------------------------- |
| 方法框架图      | 模块职责、数据流、控制流、创新模块位置    |
| 系统架构图      | 组件边界、调用关系、瓶颈位置、扩展方式    |
| 流程图          | 阶段顺序、输入输出变化、关键决策点        |
| 曲线图          | 横纵轴含义、趋势、拐点、稳定性、收敛/退化 |
| 柱状图/散点图   | 比较对象、尺度、公平性、是否有显著差距    |
| 热力图/注意力图 | 颜色代表什么、模式是否真的解释机制        |
| 案例图          | 输入输出、成功原因、失败模式、代表性      |
| 数据分布图      | 偏态、长尾、类别不平衡、标注问题          |

### 六、表格与结论数据解释

#### 表格解释总要求

请你按照论文顺序解释所有表格，每个表格至少要说明：

- 比较对象是谁。
- 在哪些数据集、指标、setting 下比。
- 最强基线是谁、提升在哪里。
- 有没有代价、波动、不公平比较或不显著提升。
- 这张表到底支撑哪一个结论。

#### 表格完整解释模板

```markdown
#### Table <编号>：<表题或自拟短标题>

**表在比较什么**  
该表比较了 <模型/系统/方法> 在 <数据集 / 场景> 上的 <指标>，setting 为 <训练/推理/资源条件>。

**主要结果**  
最强或最关键的对照对象是 <baseline>。本文方法在 <指标/数据集> 上达到 <结果>，相对其提升 <幅度>。

**结果意味着什么**  
这说明 <创新点 / 机制 / 系统设计> 在 <某类条件> 下确实带来了 <收益>，尤其是 <哪个结果最关键>。

**需要警惕的地方**  
需要注意 <是否依赖更多参数/数据/算力/人工规则>，以及 <是否有下降项 / 是否缺少更强基线 / 是否没有显著性检验>。

**我对结论强度的判断**  
这张表对“<具体结论>”的支撑力度是 <强 / 中 / 弱>，因为 <原因>。

**组会讲法**  
“Table <编号> 不是简单地说明它更高，而是在说明 <结论>。最有价值的对比是和 <baseline> 比，因为两者最接近。这里的提升主要集中在 <位置>，但它也付出了 <代价>，所以结论边界是 <边界>。”
```

#### 表格压缩解释模板

```markdown
- **Table <编号>**：在 <数据集/指标/setting> 下比较 <对象>。最关键结果是 <结果>，支撑 <结论>；但仍要注意 <代价/缺口/公平性问题>。
```

#### 按表格类型补充关注点

| 表格类型      | 额外要点                                   |
| ------------- | ------------------------------------------ |
| 主实验表      | 最强基线、主提升、跨数据集一致性           |
| 消融表        | 哪个模块真正有效，是否存在冗余             |
| 泛化/鲁棒性表 | 分布偏移、低资源、跨域、攻击条件下是否成立 |
| 效率表        | 参数量、FLOPs、延迟、吞吐、显存、成本      |
| 人评表        | 标注协议、样本量、一致性、公平性           |
| 数据统计表    | 数据规模、类别分布、标注质量、偏差         |

### 七、公式解释

#### 公式解释总要求

请你按照论文顺序解释所有公式，每个公式至少要说明：

- 优先检查该公式是否有对应图片、截图、局部裁图或论文原图中的公式区域；如果有，先展示对应图片，再在下方进行公式分析。
- 如果缺少公式对应图片，则回退为展示 LaTeX 格式公式，然后在下方继续进行公式分析。
- 公式在方法/系统中的位置。
- 符号各表示什么。
- 它本质上在计算、约束、优化或证明什么。
- 为什么需要这个公式，而不是更简单的替代方案。
- 它和创新点或实验提升的联系是什么。

#### 公式完整解释模板

```markdown
#### Equation <编号>：<公式名称或自拟短标题>

![Equation <编号> 对应图片](assets/<equation-image-file>)

> 如果没有对应图片，则改为使用 LaTeX 公式：
>
> $$
> <latex equation>
> $$

**公式所在位置**  
该公式出现在 <方法阶段 / 训练目标 / 推理决策 / 理论证明> 中，用来处理 <问题>。

**符号解释**

- <符号 1>：<含义>
- <符号 2>：<含义>
- <下标/上标/集合/概率项>：<含义>

**公式本质在做什么**  
从直觉上看，这个公式是在 <打分 / 选择 / 约束 / 优化 / 聚合 / 证明>，目标是 <一句话解释>。

**为什么这样设计**  
作者引入这个公式，是因为如果只用 <简单替代方案>，会遇到 <问题>；而当前写法试图解决 <限制>。

**和结果的关系**  
它和 <创新点 / 消融结果 / 主实验提升 / 理论结论> 直接相关，因为 <原因>。

**可能的问题或边界**  
这里需要留意 <符号定义是否完整 / 维度是否一致 / 假设是否过强 / 实现是否依赖额外技巧>。

**组会讲法**  
“Equation <编号> 可以把它理解成在做 <动作>。里面最关键的量是 <关键符号>，它控制 <作用>。作者写成这样是为了避免 <旧问题>，所以它对应的创新意义在于 <意义>。”
```

#### 公式压缩解释模板

```markdown
- **Equation <编号>**：优先展示对应公式图片；若缺失则展示 LaTeX 公式。它位于 <阶段>，本质是在 <做什么>。关键符号 <符号> 控制 <作用>，它服务于 <创新点/结论>，但前提是 <边界或假设>。
```

#### 按公式类型补充关注点

| 公式类型               | 额外要点                       |
| ---------------------- | ------------------------------ |
| 目标函数 / 损失函数    | 优化什么、各项作用、权重意义   |
| 打分 / 检索 / 选择公式 | 排序依据、决策条件、阈值来源   |
| 表示学习公式           | 表示如何构造、对齐、更新、约束 |
| 概率 / 推断公式        | 假设条件、变量关系、推断流程   |
| 系统代价公式           | 时间、空间、通信、吞吐、复杂度 |
| 定理 / 证明公式        | 条件、结论、证明直觉、适用边界 |

## 八、证据链与可信度

把论文主张和证据串起来，不要把图、表、公式分散写完就结束。

建议至少给一个证据链表：

| 主张    | 支撑证据                          | 证据强度             | 可能缺口           |
| ------- | --------------------------------- | -------------------- | ------------------ |
| <claim> | <figure/table/formula/experiment> | <strong/medium/weak> | <missing evidence> |

判断证据强度时重点看：

- 主实验是否覆盖关键场景。
- 消融是否真的证明模块必要。
- 可视化是否只是展示性材料。
- 泛化、鲁棒性、效率、失败案例是否足够。
- 作者是否把“观察到的现象”和“推测出的机制”混在一起。

## 九、与相关工作的对比与联系

这一节不是罗列参考文献，而是说明这篇论文在研究路线里的位置。

至少从四个层次来写：

- 问题层：它和哪些工作在解决同一个问题。
- 方法层：它继承、替换、组合或反对了哪些路线。
- 证据层：实验设置是否和已有工作公平可比。
- 迁移层：这套思路是否能迁移到相邻任务、组内课题或真实系统。

可选表格：

| 路线 / 工作    | 核心思路 | 主要局限     | 与本文关系                             | 比较是否充分     |
| -------------- | -------- | ------------ | -------------------------------------- | ---------------- |
| <related line> | <idea>   | <limitation> | <inherit / improve / differ / combine> | <yes/no/partial> |

## 十、优点、局限与讨论问题

优点不要空泛，必须落到证据。

局限也不要写“有待进一步研究”这种套话，要具体到：

- 缺了哪些基线、数据、场景、消融、显著性检验、效率分析或失败案例。
- 哪些结论只在特定规模、模型、数据或假设下成立。
- 是否存在工程成本、部署难度、可复现性问题。

讨论问题建议围绕：

- 创新是否真的来自核心方法。
- 证据是否足以支撑作者的强结论。
- 比较是否公平充分。
- 思路是否可以迁移。
- 如果继续做，最值得补哪块实验或机制。

## 十一、5-8 分钟汇报提纲

默认建议包含：

1. 问题和动机
2. 一句话核心想法
3. 方法总览图
4. 关键机制或关键公式
5. 最重要的主实验表
6. 一个最能解释机制的图/消融
7. 局限与讨论问题

每页都尽量给一句口头讲法，而不是只列标题。

## 最终自检清单

交付前检查：

- 是否覆盖了所有 Figure / Table / 关键 Equation 编号。
- 图片、表格、公式三节是否分别严格按原论文首次出现顺序排列。
- 是否每一项都给出了解释，而不是只复述 caption。
- 是否每个创新点都能找到对应证据。
- 是否区分了强证据、弱证据和待核对项。
- 是否写出了边界、缺口和讨论点。
- 是否已对高频专业英文短语做首次中译，并保持译名一致。
- 是否能被直接拿去做 5-8 分钟组会分享。
